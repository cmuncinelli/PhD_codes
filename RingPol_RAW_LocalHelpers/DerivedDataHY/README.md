# DerivedDataHY

**Hyperloop derived-data download and analysis framework for the RingPol analysis.**

This folder lives inside the git repository at:
```
PhD_codes/RingPol_RAW_LocalHelpers/DerivedDataHY/
```
It contains every script needed to go from a finished Hyperloop train run all the
way to a `ConsumerResults_*.root` file ready for plotting — plus a manual registry
that gives each train run a human-readable name and a stable storage path.

---

## Prerequisites — O2 environment

All scripts that communicate with the ALICE grid (`alien_ls`, `alien_cp`) and
the merger tool (`hadd`) are provided by the O2/O2Physics software environment.
**This environment must be loaded before running any script in this framework.**

Load it with:
```bash
alienv enter O2Physics/latest
```
or, if you use a pinned tag:
```bash
alienv enter O2Physics/v<TAG>-1
```

You can verify the environment is active by checking that the tools are
reachable:
```bash
which alien_ls    # should print a path inside the alienv prefix
which alien_cp
which hadd
```

If `alien_ls` or `alien_cp` are missing, the download steps will fail
immediately. If `hadd` is missing, the merge step will fail after all files
have already been downloaded — so it is worth checking upfront.

> **Tip:** if you open a new terminal mid-workflow (e.g. after a download
> finishes overnight), remember to re-enter the alienv before running
> `merge_analysis_results.sh` or `runDerivedDataConsumer_HY.sh`.

## 1. Repository layout

```
DerivedDataHY/
├── train_registry.conf                    # Manual registry: maps train run IDs to folder names
├── init_wagon_structure.sh                # Creates the storage folder tree for all registered runs
├── download_hyperloop.sh                  # Main download orchestrator (call this one)

└── convert_list_of_paths_XML_legacy.sh    # Legacy: extracts LFNs from MonALISA XML collections
```

## 2. Storage layout (outside this repo)

All data lives on the local 4 TB SSD under `~/RingPol/`. The folder tree mirrors
the registry: one folder per dataset, one subfolder per wagon configuration.

```
~/RingPol/
├── LHC25ae_pass2/
│   └── ITSandTPC_min3ITS/                  # train run 634083
│       ├── alien_paths.txt                 # <- you fill this in (paste from HY)
│       ├── aod_collections/                # <- legacy XML workflow only
│       ├── AO2Ds/                          # <- AO2D_*.root files downloaded here
│       ├── DownloadListFromAlien.txt       # auto-generated
│       ├── DownloadListAnalysisResults.txt # auto-generated
│       ├── input_data_storage.txt          # auto-generated (O2 input list)
│       ├── AnalysisResults_merged.root     # auto-generated by hadd
│       ├── dpl-config-*.json               # your analysis configs (placed manually)
│       └── results_consumer/
│           ├── ConsumerResults_*.root
│           ├── logs/
│           └── used_configs/
└── LHC25ae_pass2_small/
    ├── ITSandTPC_min3ITS/                  # train run 634033
    ├── UsingTOF/                           # train run 634032
    ├── UsingTOF_min3ITS/                   # train run 634031
    └── ITSandTPC/                          # train run 634030
```

> **Rule of thumb:** everything auto-generated by the scripts lives *inside* the wagon
> folder. Everything you provide manually (`alien_paths.txt`, dpl-configs) is also
> placed inside the wagon folder. The wagon folder is self-contained and portable.

## 4. Workflow A — Starting a new wagon analysis from scratch

Follow these steps **in order** every time you register a brand-new train run.

---

### Step 1 — Register the train run

Open [`train_registry.conf`](train_registry.conf) and add one line:

```
# TRAIN_RUN_ID  DATASET_NAME          WAGON_SHORTNAME    STORAGE_BASE
634083          LHC25ae_pass2         ITSandTPC_min3ITS  /home/users/cicerodm/RingPol
```

Fields are whitespace-separated. Lines starting with `#` are ignored.
Choose a `WAGON_SHORTNAME` that is short and unambiguous
(e.g. `ITSandTPC_min3ITS`, `UsingTOF`).

---

### Step 2 — Scaffold the folder structure

```bash
./init_wagon_structure.sh
```

Reads the registry and creates every required subdirectory for **all**
registered runs. Also creates a blank `alien_paths.txt` in each new wagon
folder. Safe to re-run — existing folders and files are never overwritten.

---

### Step 3 — Fill in `alien_paths.txt`

In the Hyperloop interface, go to the train run page, find the output path
list, and copy the comma-separated AliEN directory paths. Paste them
as-is into the wagon's `alien_paths.txt`:

```
/alice/cern.ch/user/a/alihyperloop/jobs/0438/hy_4388842,/alice/cern.ch/user/a/alihyperloop/jobs/0438/hy_4388841,...
```

One path per line, comma-separated on one line, or any mix — the script
handles all formats.

> There is no script for this step. The path list must be copied manually
> from the HY interface.

---

### Step 4 — Download everything

> **Requires the O2 environment.** Make sure `alienv enter O2Physics/latest`
> has been run in the current terminal before proceeding.

```bash
./download_hyperloop.sh 634083
```

This calls the helper scripts in sequence:

1. [`convert_alien_paths.sh`](convert_alien_paths.sh) — `alien_ls` each path → builds `DownloadListFromAlien.txt` and `DownloadListAnalysisResults.txt`
2. [`download_files_without_timeout.sh`](download_files_without_timeout.sh) — downloads `AO2D_*.root` into `AO2Ds/`
3. [`download_files_without_timeout.sh`](download_files_without_timeout.sh) — downloads `AnalysisResults_*.root` into `temp_analysis_results/`
4. [`gen_input_paths.sh`](gen_input_paths.sh) — writes `input_data_storage.txt`
5. [`merge_analysis_results.sh`](merge_analysis_results.sh) — `hadd`s all `AnalysisResults_*.root` → `AnalysisResults_merged.root`; deletes `temp_analysis_results/`

Files already present in `AO2Ds/` are automatically skipped, so interrupted
downloads can be safely resumed by re-running the same command.

---

### Step 5 — Place your dpl-config JSON files

Copy the consumer configuration(s) you want to run into the wagon folder:

```bash
cp dpl-config-DerivedConsumer-JustLambda.json \
    ~/RingPol/LHC25ae_pass2/ITSandTPC_min3ITS/
```

---

### Step 6 — Run the consumer

> **Requires the O2 environment.** If you opened a new terminal since
> the download finished, re-enter alienv before running this step.

```bash
./runDerivedDataConsumer_HY.sh \
    ~/RingPol/LHC25ae_pass2/ITSandTPC_min3ITS \
    ~/RingPol/LHC25ae_pass2/ITSandTPC_min3ITS/dpl-config-DerivedConsumer-JustLambda.json
```

Output lands in:
```
~/RingPol/LHC25ae_pass2/ITSandTPC_min3ITS/results_consumer/
  ConsumerResults_JustLambda.root
  logs/consumer_run_JustLambda.log
  used_configs/consumer-config_JustLambda.json
```

## 5. Workflow B — Appending or updating an existing analysis

### B1 — New train run on an already-known dataset

The most common case: a new HY train run exists for a dataset you already
have a folder for.

1. Add the new run to [`train_registry.conf`](train_registry.conf).
2. Run `./init_wagon_structure.sh` to create the new wagon subfolder (existing ones are untouched).
3. Follow **Steps 3 → 6** from Workflow A for the new run ID.

---

### B2 — Re-running the consumer with a different dpl-config

You already have the AODs downloaded and just want to try a different
analysis configuration.

```bash
./runDerivedDataConsumer_HY.sh \
    ~/RingPol/LHC25ae_pass2/ITSandTPC_min3ITS \
    ~/RingPol/LHC25ae_pass2/ITSandTPC_min3ITS/dpl-config-DerivedConsumer-BothHyperons.json
```

Because the output filename encodes the consumer config suffix
(`ConsumerResults_BothHyperons.root`), results from different configs
coexist in `results_consumer/` without overwriting each other.

---

### B3 — Downloading additional files for an existing wagon

If the HY train run was extended and new output directories are available:

1. Update `alien_paths.txt` with the full new list of paths (including both old and new).
2. Re-run `./download_hyperloop.sh <TRAIN_RUN_ID>`.

The download script skips files already present in `AO2Ds/`, so only the
genuinely new files are fetched. Both LFN lists, `input_data_storage.txt`,
and `AnalysisResults_merged.root` are all regenerated from scratch to
reflect the complete updated set.

---

### B4 — Updating FRAMEWORK_DIR after a repo move

The only hardcoded absolute path in this framework is the `FRAMEWORK_DIR`
variable at the top of [`download_hyperloop.sh`](download_hyperloop.sh) and
[`init_wagon_structure.sh`](init_wagon_structure.sh).
If you move or rename the `DerivedDataHY` folder, update that one line in
each of those two files.

## 6. Script reference

### [`train_registry.conf`](train_registry.conf)

The single source of truth for all registered train runs. Maintained manually.

```
# TRAIN_RUN_ID  DATASET_NAME          WAGON_SHORTNAME      STORAGE_BASE
634083          LHC25ae_pass2         ITSandTPC_min3ITS    /home/users/cicerodm/RingPol
634033          LHC25ae_pass2_small   ITSandTPC_min3ITS    /home/users/cicerodm/RingPol
634032          LHC25ae_pass2_small   UsingTOF             /home/users/cicerodm/RingPol
634031          LHC25ae_pass2_small   UsingTOF_min3ITS     /home/users/cicerodm/RingPol
634030          LHC25ae_pass2_small   ITSandTPC            /home/users/cicerodm/RingPol
```

---

### [`init_wagon_structure.sh`](init_wagon_structure.sh)

```bash
./init_wagon_structure.sh
```

Reads the registry and creates the following tree for every entry:
```
<WAGON>/
  alien_paths.txt          <- blank; fill with AliEN paths from HY
  aod_collections/         <- legacy XML workflow only
  AO2Ds/
  results_consumer/
  results_consumer/logs/
  results_consumer/used_configs/
```
Safe to re-run at any time. A hidden `.initialized` sentinel records the
creation timestamp; subsequent runs skip already-initialized wagons.

---

### [`download_hyperloop.sh`](download_hyperloop.sh)

```bash
./download_hyperloop.sh <TRAIN_RUN_ID>
```

Main entry point. Looks up the train run in the registry, verifies
`alien_paths.txt` is non-empty, then calls the five helper scripts in
order. No assumption is made about the current working directory.

---

### [`convert_alien_paths.sh`](convert_alien_paths.sh)

```bash
./convert_alien_paths.sh <WORK_DIR>
```

Called automatically by the orchestrator. Reads `<WORK_DIR>/alien_paths.txt`
(comma-separated, one-per-line, or mixed), runs `alien_ls` on each path to
enumerate one-level-deep subdirectories, and writes:

- `DownloadListFromAlien.txt` — one `AO2D.root` LFN per line
- `DownloadListAnalysisResults.txt` — one `AnalysisResults.root` LFN per line

Expected grid structure (confirmed, one level deep):
```
<alien_path>/001/AO2D.root
<alien_path>/001/AnalysisResults.root
<alien_path>/002/AO2D.root
...
```

---

### [`download_files_without_timeout.sh`](download_files_without_timeout.sh)

```bash
./download_files_without_timeout.sh <LIST_FILE> <OUTPUT_DIR> <FILE_PREFIX>
```

Called twice by the orchestrator — once for AO2Ds and once for
AnalysisResults. Downloads every LFN in `LIST_FILE` into `OUTPUT_DIR`,
naming files `<FILE_PREFIX>_1.root`, `<FILE_PREFIX>_2.root`, etc.
Already-present files are skipped. Timeout is disabled; `alien_cp`
manages its own retries.

---

### [`merge_analysis_results.sh`](merge_analysis_results.sh)

```bash
./merge_analysis_results.sh <WORK_DIR>
```

Called automatically by the orchestrator. Runs `hadd -f` on all
`AnalysisResults_*.root` files in `<WORK_DIR>/temp_analysis_results/`,
writing the merged output to `<WORK_DIR>/AnalysisResults_merged.root`.
On success, `temp_analysis_results/` is deleted. On failure it is
preserved for inspection, and the script exits non-zero.

---

### [`gen_input_paths.sh`](gen_input_paths.sh)

```bash
./gen_input_paths.sh <WORK_DIR> <N>
```

Called automatically by the orchestrator. Generates
`<WORK_DIR>/input_data_storage.txt` containing `N` lines of the form:
```
file:/home/users/cicerodm/RingPol/<dataset>/<wagon>/AO2Ds/AO2D_1.root
file:/home/users/cicerodm/RingPol/<dataset>/<wagon>/AO2Ds/AO2D_2.root
...
```
The `file:` prefix is required by O2 to distinguish local paths from
AliEN grid LFNs.

---

### [`runDerivedDataConsumer_HY.sh`](runDerivedDataConsumer_HY.sh)

```bash
./runDerivedDataConsumer_HY.sh <WORK_DIR> <consumer_config.json>
```

Runs `o2-analysis-lf-lambdajetpolarizationionsderived` on all
`AO2Ds/AO2D_*.root` files. The output filename is derived from the
consumer config name by stripping the `dpl-config-DerivedConsumer-`
prefix, so different configs produce distinct output files and never
overwrite each other.

| Path | Contents |
|---|---|
| `results_consumer/ConsumerResults_<SUFFIX>.root` | Analysis output |
| `results_consumer/logs/consumer_run_<SUFFIX>.log` | Full O2 stdout/stderr |
| `results_consumer/used_configs/consumer-config_<SUFFIX>.json` | Config snapshot |

O2 scratch files are written to `temp_consumer_stage/` inside `WORK_DIR`
and cleaned up automatically on exit.

---

### [`convert_list_of_paths_XML_legacy.sh`](convert_list_of_paths_XML_legacy.sh)

```bash
./convert_list_of_paths_XML_legacy.sh <WORK_DIR>
```

Legacy script for the old XML-based workflow. Scans
`<WORK_DIR>/aod_collections/*.xml`, extracts every `lfn="..."` attribute,
and writes the paths to `<WORK_DIR>/DownloadListFromAlien.txt`.
Not called by the main orchestrator. Kept for reference and for any
dataset that was originally collected via MonALISA XML exports.
